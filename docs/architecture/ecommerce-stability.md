# 电商生图稳定性架构

## 当前上线边界

- 受邀账号在服务端校验，页面提示不是安全边界。
- 每次电商生成建立 SQLite 任务记录，保存归属、进度、结果和失败原因。
- 全部外部图片请求经过一个进程级并发池，默认同时执行 3 个、最多排队 24 个。
- 产品图以 3–5 张多角度实拍为推荐输入，最多取 8 张；参考图用于视觉分析，最多取 48 张。
- 受限于当前图片网关的单图输入协议，服务会把产品图优先、少量参考图随后合成为视觉锚点，替代旧版“忽略产品图、只用第一张参考图”的行为。
- 生成完成后，服务会将上游临时图片下载、校验并以内容哈希保存为本站 `/api/generated-assets/*` 资源；作品库只保存稳定地址，避免画布因第三方签名 URL 过期而全部“加载失败”。
- 作品数据统一写入 SQLite 的完整载荷字段，保留电商套图、SKU、提示词和结果等扩展信息；旧 `works.json` 仅在数据库为空时一次性迁移。
- 电商首页使用紧凑的“产品图 × 参考图”工作台：产品图按正面、侧面、背面、细节、尺度引导补齐，参考图可横向浏览与批量添加，不再挤压页面布局。

## 下一阶段拆分

`server/index.mjs` 仍承担旧接口兼容。后续按以下边界拆出，而不是再次整体重写：

1. `ecommerceJobService`：任务创建、恢复、结果读取。
2. `ecommerceGenerationService`：输入解析、视觉分析、提示词组装、图片池调用。
3. `authService`：受邀名单、验证码发送和会话签发。
4. 独立 worker：当内测扩展到多账号时，从 Web 进程移出长时生图执行，并接入 Redis/托管队列。
5. 对象存储：当生成资产或用户上传量持续增长时，将 `generated-assets` 和上传文件迁移至 COS/S3，并通过 CDN 提供稳定读取；数据库仍只保存对象键与元数据。

在单机内测阶段保留 SQLite + 受控进程池，可避免引入 Redis 后新的运维单点；扩展到多实例前必须完成第 4 步，扩展到高并发图片资产前应完成第 5 步。
